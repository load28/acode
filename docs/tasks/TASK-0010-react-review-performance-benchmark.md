## 목표

MCP `review_code`/`check_code`로 실제 React(TSX) 코드를 리뷰할 때 걸리는
시간을 **실제 오픈소스 저장소**를 입력으로 측정한다. 합성 코드가 아니라
실코드 기준으로 (1) 결정적 구간(룰 검증·검색)과 (2) LLM 구간을 분리해
병목이 어디인지 수치로 답한다. 재현 가능하도록 벤치마크 스크립트를
저장소에 남긴다.

## 결정 사항

- 결정: 벤치 입력은 합성 코드가 아닌 실저장소(excalidraw). 이유: 사용자 요구
  ("실제 레포로 테스트"); 실코드의 파일 크기 분포·문법 다양성(JSX 승격,
  대형 아이콘 파일, 테스트 파일)이 합성으로는 재현 안 됨. 저장소 자체는
  커밋하지 않고 스크립트가 경로를 argv로 받게 함.
- 결정: LLM 벤치는 파일 2개 × 2회만. 이유: 호출당 24~91초 + 비용;
  분산 파악에는 2회면 충분하고 병목 판정(LLM ≈ 100%)은 1회로도 명백.
- 관찰: 인덱싱된 스토어(패턴 1,439~2,089개)에서는 retrieval이 top_k=8
  히트를 더 채워 프롬프트가 커지고(7k자 → 21k자) LLM 시간도 늘어남
  (소형 파일 24~29s → 44~69s). 리뷰 지연에는 스토어 크기가 간접 영향.
- 관찰: 683줄 파일 1차 측정에서 LLM 수정본이 기계 재검증에 실패
  (`fix_verified=False`) 후 재실행에선 통과 — 프로즈 대신 기계 판정을
  싣는 설계가 실제로 변별력 있음을 확인.

## 검증 결과

입력: excalidraw@master 얕은 클론, TSX 295파일 / 92,612줄. 룰 14개 적용
(tsx 4 + typescript 상속 11 − override 1).

**check_code 경로 (LLM 없음), 전체 295파일** — `bench_review_mech.py`:

- 평균 13.1~16.5ms, 중앙값 4.5~5.7ms, p90 28.8~36.7ms, 최대 630~705ms
  (14,124줄 `App.tsx`)
- 처리량 19,000~24,000 줄/초 — 저장소 전체 스캔 3.9~4.9초, 위반 736건 검출

**retrieval (top_k=8, AST 유사도 포함)**:

- 시드 스토어(33개 컨벤션): 중앙값 1.6~3.3ms
- `index_codebase` 후(2,122개): 중앙값 70.8~72.2ms, 최대 199~391ms
  (인덱싱 자체: 500파일 → 2,089패턴, 9.2초)

**전체 `review_code` (claude CLI 프로바이더), 각 2회** — `bench_review_full.py`:

| 파일 | 스토어 | 총 소요 | LLM 비중 | 기계 구간 |
|---|---|---|---|---|
| 111줄 (DropdownMenuContent.tsx) | 시드만 | 24.2s / 29.3s | ~100% | ≤0.04s |
| 111줄 (〃) | 인덱싱됨 | 43.6s / 69.2s | ~100% | ≤0.21s |
| 683줄 (flowchart.test.tsx) | 시드만 | 81.1s / 79.2s | ~100% | ≤0.07s |
| 683줄 (〃) | 인덱싱됨 | 90.8s / 85.0s | ~100% | ≤0.18s |

claude CLI는 한 단어 응답에도 ~8.7초의 고정 오버헤드가 있음.

**결론**: 리뷰 지연은 사실상 전부 LLM 대기 시간. 결정적 검증은 중앙값
~5ms라 CI에서 저장소 전체(295파일)를 5초 내에 돌릴 수 있고, MCP
`review_code`는 호출당 24~91초. 위반 검출만 필요하면 `check_code`
(LLM 없음), 수정본 생성이 필요할 때만 `review_code`가 올바른 사용 패턴.
